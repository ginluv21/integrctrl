# integrctrl - полный разбор проекта

Вариант 2.5. Утилита командной строки на языке Си для Linux.
Назначение: контроль целостности файлов через MD5-хеши.

---

## Структура файлов

```
integrctrl/
├── src/
│   ├── main.c          точка входа
│   ├── integrctrl.c    разбор аргументов, обход папок, режимы -s и -c
│   ├── check.c         сравнение старых и новых записей
```

## Как работает программа в целом

Программа запускается с флагами командной строки:

```
integrctrl -s [-r] -f <файл_базы> <директория>   - сохранить хеши
integrctrl -c [-r] -f <файл_базы> [директория]   - проверить целостность
```

**Режим -s (save):**
1. Обходит указанную директорию, для каждого файла считает MD5-хеш.
2. Все найденные файлы и папки записывает в бинарный файл базы.

**Режим -c (check):**
1. Читает базу из файла.
2. Снова обходит директорию (или ту, что была записана в базе).
3. Сравнивает старые и новые данные, выводит статус каждого объекта.

## Структуры данных

### file_rec (include/integrctrl.h)

```c
typedef struct file_rec {
    int id;               // уникальный номер записи (начинается с 1)
    char name[256];       // имя файла или папки (только имя, не полный путь)
    int type;             // TYPE_DIR=0 или TYPE_FILE=1
    int parent_id;        // id родительской папки; у корневой записи parent_id=0
    unsigned char md5[16]; // 16 байт MD5-хеша; для папок заполнено нулями
} file_rec;
```

Интересный момент - полный путь нигде не хранится. Хранится только имя и ссылка
на родителя. Чтобы получить полный путь, нужно пройти по цепочке parent_id вверх.
Это экономит место и позволяет легко переименовывать папки в будущем.

Корневая запись - это запись с parent_id=0. Её имя - это путь к директории,
которую сканировали (например "/home/user/project"). Все остальные записи имеют
parent_id, указывающий на их родителя.

### prog_opts (include/integrctrl.h)

```c
typedef struct {
    int mode;        // 's' или 'c' (char хранится как int)
    int recursive;   // 1 если был флаг -r, иначе 0
    char *db_file;   // указатель на строку из argv[] с именем файла базы
    char *target_dir; // указатель на строку из argv[] с именем директории
} prog_opts;
```

Указатели db_file и target_dir НЕ копируют строку - они просто указывают на
элементы массива argv[]. Это нормально, потому что argv[] существует всё время
работы программы.

### db_header (include/db.h)

```c
typedef struct {
    unsigned int magic;        // 0x41634463 = "AcDc" в ASCII
    unsigned int record_count; // сколько file_rec записано в файле
    int recursive;             // был ли флаг -r при сохранении
    time_t saved_at;           // время сохранения (секунды с 01.01.1970)
} db_header;
```

Это заголовок бинарного файла базы. Он записывается первым, перед всеми
file_rec. Magic number нужен чтобы убедиться, что файл действительно является
базой integrctrl, а не каким-то другим файлом.

time_t - это тип из <time.h>, обычно 8 байт (int64_t). Хранит количество секунд
прошедших с 1 января 1970 года ("эпоха Unix"). Функция time(NULL) возвращает
текущее время в этом формате.

### vector_t (include/vector.h)

```c
typedef struct {
    file_rec **data;   // массив указателей на file_rec
    file_rec **res;    // резервный буфер для быстрого расширения
    size_t len;        // текущее количество элементов
    size_t cap;        // текущая ёмкость (сколько можно хранить без расширения)
    int owns_data;     // 1 = free каждый элемент при уничтожении, 0 = не трогать
} vector_t;
```

data - это массив УКАЗАТЕЛЕЙ, то есть каждый элемент вектора - это указатель
на file_rec, выделенный отдельно через malloc(). Сам вектор не хранит
file_rec напрямую, только адреса.

Поле res (резерв) - это второй буфер заранее выделенной памяти. Когда основной
буфер data заканчивается, вектор переключается на res (он уже выделен, не надо
ждать malloc) и одновременно выделяет новый res. Это делает расширение быстрее.

## Файл src/main.c

```c
#include "integrctrl.h"
int main(int argc, char *argv[]) {
    prog_opts opts;
    parse_args(argc, argv, &opts);
    if (opts.mode == 's')
        run_save(&opts);
    else
        run_check(&opts);
    return 0;
}
```

Простая точка входа. parse_args проверяет аргументы и заполняет opts.
Дальше вызывается нужный режим.

## Файл src/integrctrl.c

### parse_args

```c
void parse_args(int argc, char *argv[], prog_opts *opts)
```

Использует стандартную функцию getopt() из <unistd.h>.

getopt() вызывается в цикле. Каждый вызов разбирает один флаг из argv[].
Строка "scrf:" означает:
- `s` - флаг без аргумента
- `c` - флаг без аргумента
- `r` - флаг без аргумента
- `f:` - флаг с аргументом (двоеточие = аргумент обязателен)

После цикла getopt() глобальная переменная optind содержит индекс первого
не-флагового аргумента в argv[]. Так получаем target_dir (директорию).

Если обязательные параметры не указаны - вызывается print_usage() и программа
завершается через exit(EXIT_FAILURE).

### scan_directory

```c
void scan_directory(const char *dir_path, int parent_id, int *next_id,
                    vector_t *vec, int recursive)
```

Функция обхода директории.

`opendir(dir_path)` - открывает директорию, возвращает указатель DIR*.
`readdir(dir)` - читает следующий элемент директории, возвращает struct dirent*.
`closedir(dir)` - закрывает директорию.

struct dirent содержит:
- d_name - имя файла/папки
- d_type - тип: DT_REG (обычный файл) или DT_DIR (директория)

Для каждого найденного объекта создаётся file_rec через malloc(). Заполняются
поля: id, name, type, parent_id. Для файлов вычисляется MD5 (через md5File).
Запись добавляется в вектор через vec_push().

Если объект - папка и recursive=1, функция вызывает сама себя (рекурсия).
При этом parent_id новых записей = id текущей папки.

next_id - указатель на счётчик. Передаётся по указателю чтобы рекурсивные
вызовы продолжали нумерацию, а не начинали заново.

Файлы `.` и `..` пропускаются - это служебные ссылки "текущая папка"
и "родительская папка", они всегда есть в любой директории.

Символические ссылки (DT_LNK) и другие типы тоже пропускаются -
обрабатываются только обычные файлы и папки.

### run_save

```c
void run_save(prog_opts *opts)
```

1. Создаёт вектор через vec_create(10).
2. Создаёт корневую запись (parent_id=0, имя = путь к директории).
3. Вызывает scan_directory для обхода содержимого.
4. Автосоздание папки для базы: берёт путь к базе, ищет последний `/`,
   обрезает имя файла - остаётся путь к папке. mkdir() создаёт её (если есть).
5. Вызывает db_save() для записи базы.
6. Выводит количество объектов.
7. Уничтожает вектор через vec_destroy().

Автосоздание папки - это `strrchr(dir_buf, '/')` для поиска последнего слеша.
Если слеш найден, всё после него обрезается (`*slash = '\0'`) и вызывается
`mkdir(dir_buf, 0755)`. Если папка уже есть - mkdir вернёт ошибку,
но это нормально, мы её игнорируем.

### run_check

```c
void run_check(prog_opts *opts)
```

1. Загружает базу через db_load(), получает old_vec, db_recursive, saved_at.
2. Из первого элемента вектора берёт имя корневой директории (db_root).
3. Определяет target_dir: если пользователь указал папку в аргументах - берёт её,
   иначе берёт из базы.
4. Если указанная папка отличается от корня базы - вычисляет prefix.
   Prefix - это относительный путь от корня базы до нужной подпапки.
   Например если база записана для `/home/user/proj`, а проверяем `src` -
   prefix будет "src".
5. Форматирует дату сохранения базы через strftime().
6. Создаёт new_vec, сканирует target_dir.
7. Вызывает check_integrity() - сравнение.
8. Выводит разделители, статистику, количество объектов.
9. Уничтожает оба вектора.

Важная деталь: `int recursive = opts->recursive` - рекурсия берётся только из
аргументов командной строки. Значение db_recursive из базы читается,
но для сканирования НЕ используется. Это сделано намеренно: пользователь должен
явно указать -r если хочет рекурсивную проверку.

Форматирование даты:
```c
struct tm *tm_info = localtime(&saved_at);
strftime(time_buf, sizeof(time_buf), "%Y-%m-%d %H:%M:%S", tm_info);
```
localtime() переводит time_t (секунды от эпохи) в структуру tm с полями:
год, месяц, день, часы, минуты, секунды.
strftime() форматирует tm в строку по шаблону.

## Файл src/check.c

### find_by_id

```c
static file_rec *find_by_id(vector_t *vec, int id)
```

Линейный поиск по вектору: перебирает все элементы пока не найдёт запись с
нужным id. Возвращает указатель на неё или NULL если не нашёл.

Ключевое слово `static` означает: функция видна только внутри файла check.c.
Это нормальная практика для вспомогательных функций.

### build_path

```c
static void build_path(vector_t *vec, file_rec *rec, char *buf, size_t size)
```

Строит полный относительный путь к записи.

Поскольку в file_rec хранится только имя (не полный путь), нужно идти по цепочке
parent_id вверх до корня и собирать путь.

Логика:
- Если parent_id == 0, значит это корневая запись. Путь пустой, buf = "".
- Если родитель корневой (parent_id родителя == 0), то путь = просто имя файла.
- Иначе: сначала строим путь родителя (рекурсия), потом дописываем "/" и имя.

Результат: для файла с путём `src/main.c` в буфере окажется "src/main.c".
Для самой папки src - "src".

Пример дерева:
```
id=1 name="/home/user/proj" parent_id=0   (корневая)
id=2 name="src"             parent_id=1   (папка)
id=3 name="main.c"          parent_id=2   (файл)
```
build_path для id=3: родитель id=2 (src), его родитель id=1 (корень, parent_id=0).
Значит путь для id=2 это просто "src", добавляем "/" и "main.c" → "src/main.c".

### find_by_path

```c
static file_rec *find_by_path(vector_t *vec, const char *path)
```

Для каждой не-корневой записи в векторе строит путь через build_path и сравнивает
с нужным. Возвращает первую совпавшую запись или NULL.

### md5_equal

```c
static int md5_equal(unsigned char *a, unsigned char *b)
```

Сравнивает два MD5-хеша побайтово. MD5 - это всегда 16 байт.
Возвращает 1 если хеши одинаковы, 0 если отличаются.

### check_integrity

```c
void check_integrity(vector_t *old_vec, vector_t *new_vec, const char *prefix,
                     int stats[4], int recursive)
```

Основная функция сравнения. Принимает:
- old_vec - данные из базы
- new_vec - данные только что отсканированные
- prefix - подпапка для фильтрации (или NULL)
- stats - массив из 4 чисел: [0]=OK [1]=CHANGED [2]=MISSING [3]=NEW
- recursive - 1 если нужно показывать вложенные файлы

**Первый цикл** - проходим по записям из базы (old_vec):
1. Пропускаем корневую запись (parent_id == 0).
2. Строим путь записи в old_vec через build_path → old_path.
3. Если задан prefix - фильтруем: оставляем только записи внутри этой подпапки.
   Также пропускаем саму папку prefix (нам нужно только её содержимое).
   Вычисляем search = часть пути после prefix + '/'.
4. Если не рекурсивно (recursive=0) и в search есть '/' - пропускаем
   (это файл во вложенной папке, нам он не нужен).
5. Ищем эту же запись в new_vec через find_by_path(new_vec, search).
6. Если не нашли - MISSING (красный).
7. Если нашли но хеши разные (только для файлов) - CHANGED (жёлтый).
8. Иначе - OK (зелёный).

**Второй цикл** - проходим по записям из нового скана (new_vec):
1. Строим путь в new_vec → new_path.
2. Если не рекурсивно и в new_path есть '/' - пропускаем.
3. Если с prefix - формируем full_search = prefix + '/' + new_path для поиска в old_vec.
4. Ищем в old_vec. Если не нашли - это NEW (голубой).

Интересное место - для поиска NEW мы ищем в old_vec полный путь с prefix,
потому что в old_vec пути хранятся с учётом всей иерархии от корня,
а в new_vec - только относительно нового скана.

**Цветной вывод:**
```c
printf("%-23.23s  [%s]  %s%-7s\033[0m\n", search, t, color, status);
```
`\033[31m` - включить красный цвет. `\033[0m` - сбросить всё.
Числа - это коды цветов ANSI. 31=красный, 32=зелёный, 33=жёлтый, 36=голубой.
`%-23.23s` - строка левое выравнивание, минимум 23 символа, максимум 23 символа.
`%-7s` - строка левое выравнивание, минимум 7 символов.

## Файл src/db.c

### db_save

```c
int db_save(const char *path, vector_t *vec, int recursive)
```

1. Открывает файл на запись в бинарном режиме (`"wb"`).
   `b` - binary, без преобразования символов (на Linux не важно, на Windows важно).
2. Заполняет db_header: magic, record_count, recursive, saved_at = time(NULL).
3. Записывает заголовок: `fwrite(&hdr, sizeof(db_header), 1, f)`.
   Это записывает sizeof(db_header) байт одним блоком.
4. В цикле записывает каждый file_rec: `fwrite(rec, sizeof(file_rec), 1, f)`.

Файл базы - это просто последовательность байт:
```
[db_header][file_rec][file_rec][file_rec]...[file_rec]
```
Никаких разделителей, никакого текста. Чистый бинарный формат.

Возвращает 0 при успехе, 1 при ошибке.

### db_load

```c
vector_t *db_load(const char *path, int *recursive_out, time_t *saved_at_out)
```

1. Открывает файл на чтение в бинарном режиме (`"rb"`).
2. Читает заголовок: `fread(&hdr, sizeof(db_header), 1, f)`.
   fread возвращает количество прочитанных элементов. Если не 1 - ошибка.
3. Проверяет magic number. Если не совпадает - файл не наш, ошибка.
4. Записывает в out-параметры recursive и saved_at (если указатели не NULL).
5. Создаёт вектор с ёмкостью record_count (точно знаем сколько записей).
6. В цикле hdr.record_count раз:
   - Выделяет malloc(sizeof(file_rec))
   - Читает fread(rec, sizeof(file_rec), 1, f)
   - Добавляет в вектор vec_push()

Важно: каждый file_rec выделяется отдельно через malloc. Это нужно чтобы
vec_destroy() мог освободить их по одному через free().

Если на любом шаге ошибка - освобождает всё что успело выделиться и возвращает NULL.

## Файл src/vector.c

### vec_create

```c
vector_t *vec_create(size_t ini_cap)
```

Выделяет память для vector_t, для массива data (ini_cap указателей) и для
резервного буфера res (ini_cap * 1.41 указателей). Устанавливает owns_data=1.

Коэффициент 1.41 примерно равен √2. Это типичный коэффициент роста для
динамических массивов - он обеспечивает амортизированную O(1) вставку.

### vec_destroy

```c
void vec_destroy(vector_t *vec)
```

Если owns_data=1 - освобождает каждый file_rec через free().
Затем освобождает data, res, и сам vector_t.

Порядок важен: сначала элементы, потом массив указателей, потом структура.

### vec_resize (static)

```c
static void vec_resize(vector_t *vec, size_t new_cap)
```

Уменьшает массив data через realloc(). Используется при удалении элементов
когда вектор стал сильно меньше своей ёмкости (len < cap/4).

### vec_reserve (static)

```c
static int vec_reserve(vector_t *vec)
```

Вызывается когда len == cap (вектор полон).

Механизм:
1. Копируем все указатели из data в res (оба буфера существуют одновременно).
2. free(data) - освобождаем старый основной буфер.
3. data = res - резервный становится основным (без нового malloc!).
4. Выделяем новый res с ёмкостью cap * 1.41.

Интересный момент: переключение data → res происходит без malloc, потому что
res уже выделен заранее. Это ключевая идея двойного буфера.

### vec_push

```c
int vec_push(vector_t *vec, file_rec *rec)
```

Если len == cap - вызывает vec_reserve() для расширения.
Добавляет указатель в data[len], увеличивает len.

Запись `*(vec->data + vec->len) = rec` - это то же самое что `vec->data[vec->len] = rec`.
Оба варианта одинаковы в Си.

### vec_pop

```c
file_rec *vec_pop(vector_t *vec)
```

Возвращает последний элемент, уменьшает len.
Если len стало меньше cap/4 - уменьшает массив через vec_resize().
Освобождение памяти file_rec - ответственность вызывателя (функция только возвращает указатель).

### vec_get

```c
file_rec *vec_get(vector_t *vec, size_t ind)
```

Возвращает указатель на элемент по индексу с проверкой границ.

### vec_insert / vec_remove / vec_change

Стандартные операции. vec_insert сдвигает элементы вправо (vec_shr),
vec_remove - влево (vec_shl). vec_change заменяет элемент, освобождая старый.

### Итератор (vec_iter_t)

Структура хранит указатель на вектор и текущий индекс.
vec_begin() возвращает итератор на первый элемент.
vec_end() возвращает итератор за последним элементом.
vec_iter_next() сдвигает итератор на следующий элемент.
vec_isequal() сравнивает два итератора.

В текущей программе итераторы не используются, но реализованы для полноты.

## MD5-хеш - как это работает (приближённо)

MD5 - это алгоритм хеширования: принимает данные любого размера, возвращает
16 байт (128 бит). Один и тот же файл всегда даёт один и тот же хеш.
Изменение хотя бы одного байта в файле полностью меняет хеш.

Реализация взята из https://github.com/Zunawe/md5-c
Используется функция `md5File(FILE *fp, uint8_t *result)`.

Приближённая схема работы MD5:
1. Файл читается блоками по 512 бит (64 байта).
2. Каждый блок обрабатывается через 64 раунда преобразований.
3. Каждый раунд применяет нелинейную функцию (смесь AND, OR, XOR, NOT),
   прибавляет константу, и делает побитовый сдвиг.
4. Константы выбраны из таблицы abs(sin(i)) - так получаются "случайные" числа.
5. После всех блоков получается 128-битный результат - это и есть хеш.

Почему хеш меняется при изменении файла: любое изменение входных данных
проходит через нелинейные операции и лавинообразно меняет весь результат.
Это называется "лавинный эффект" (avalanche effect).

В программе хеш сравнивается побайтово в функции md5_equal().
Папки имеют хеш из нулей (memset(rec->md5, 0, 16)) - для папок
хеш не вычисляется, только для файлов.

## Формат бинарного файла базы

```
Байты 0..sizeof(db_header)-1 :  заголовок
  unsigned int magic         (4 байта)  = 0x41634463
  unsigned int record_count  (4 байта)  = количество записей
  int recursive              (4 байта)  = 0 или 1
  time_t saved_at            (8 байт)   = секунды от эпохи Unix

Затем record_count раз подряд:
  int id                     (4 байта)
  char name[256]             (256 байт)
  int type                   (4 байта)
  int parent_id              (4 байта)
  unsigned char md5[16]      (16 байт)
                             = 284 байта на запись
```

Итого для 10 объектов: 20 + 10*284 = 2860 байт.

## ANSI escape-коды - механика цветного вывода

Терминал в Linux понимает специальные последовательности байт внутри текста.
Когда терминал встречает `\033[` (байт 27, потом '['), он читает дальше
до буквы 'm' и интерпретирует как команду изменения цвета/стиля.

`\033` - восьмеричная запись числа 27 (0x1B), это символ ESC.

Таблица кодов цветов текста:
```
\033[31m  - красный    (MISSING)
\033[32m  - зелёный    (OK)
\033[33m  - жёлтый     (CHANGED)
\033[36m  - голубой    (NEW)
\033[0m   - сброс всех атрибутов (возврат к обычному цвету)
```

В коде:
```c
const char *color = "\033[32m";
printf("%s%-7s\033[0m\n", color, status);
```
Сначала печатаем управляющую последовательность включения цвета,
потом текст, потом `\033[0m` - отключаем цвет.
Если не отключить - весь последующий вывод тоже будет цветным.

Эти коды работают только в терминале. Если перенаправить вывод в файл
(`./integrctrl -c ... > out.txt`), в файле будут видны сырые байты `^[[32m`.

## Построение пути - подробный пример

Допустим сохранили папку `src` с флагом -r. Записи в базе:

```
id=1 name="src"     parent_id=0  type=DIR   (корень)
id=2 name="main.c"  parent_id=1  type=FILE
id=3 name="sub"     parent_id=1  type=DIR
id=4 name="helper.c" parent_id=3 type=FILE
```

Вызов build_path для id=4 ("helper.c"):
1. parent_id=3, не ноль. Ищем parent - id=3 ("sub").
2. parent_id id=3 равен 1, а id=1 это корень (parent_id=0).
   Значит parent_id родителя != 0, идём в рекурсию.
3. Вызываем build_path для id=3 ("sub"):
   - parent_id=1, ищем - id=1 ("src"), его parent_id=0.
   - Значит parent id=3 корневой. Копируем имя: buf = "sub".
4. Возвращаемся, buf = "sub". Добавляем '/' → "sub/". Дописываем "helper.c".
5. Итог: buf = "sub/helper.c".

В check_integrity функция find_by_path() вызывает build_path для каждой записи
и сравнивает результат с нужным путём. Это позволяет найти соответствие между
записью в базе и текущим файлом.
